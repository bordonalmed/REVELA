1. O QUE É O REVELA
================================================================================

O Revela é uma aplicação web profissional (PWA) para comparação de fotos 
antes e depois de tratamentos. É voltado para profissionais da saúde como 
médicos, dentistas, dermatologistas, cirurgiões plásticos, esteticistas, 
fisioterapeutas e clínicas estéticas.

Diferencial principal: as fotos ficam armazenadas LOCALMENTE no dispositivo 
do usuário (IndexedDB), garantindo privacidade total. Não há upload 
obrigatório para nuvem.

URL: https://www.revela.fun
Deploy: Netlify
Autenticação: Supabase
Pagamentos: Hotmart


2. TECNOLOGIAS UTILIZADAS
================================================================================

- Framework: Next.js 14.1 (App Router)
- Frontend: React 18.2 + TypeScript 5.3
- Estilização: Tailwind CSS 3.4
- Componentes UI: Radix Slot + shadcn-style + Lucide Icons
- Auth & Backend: Supabase (autenticação + entitlements)
- Validação: Zod + React Hook Form
- Imagens: react-easy-crop, react-image-crop, Canvas APIs
- PDF: jsPDF
- Deploy: Netlify (com Next.js plugin)
- PWA: Service Worker + manifest.json
- Analytics: Google Analytics 4 + Meta Pixel + Twitter + TikTok
- Internacionalização: pt-BR e en-US

5. FLUXO DE AUTENTICAÇÃO
================================================================================

1. Usuário acessa /signup → preenche email + senha
2. Supabase cria a conta (signUp)
3. Redireciona para /dashboard
4. Em páginas protegidas, verifica getSession()
5. Sem sessão → redireciona para /login
6. Logout → supabase.auth.signOut() → volta para /

Modo Demo: com NEXT_PUBLIC_DEMO_MODE=true, pula Supabase e usa sessão fake.

8. HOOKS CUSTOMIZADOS
================================================================================

useAuth(redirectToLogin?: boolean)
  - Gerencia sessão Supabase
  - Verifica getSession() e onAuthStateChange
  - Redireciona para /login se não autenticado
  - Retorna { user, loading }

usePlan()
  - Resolve plano do usuário (free/pro/premium)
  - Consulta revela_entitlements no Supabase
  - Refaz consulta quando aba ganha foco (pós-compra)
  - Retorna { userPlan, hasProAccess, hasPremiumAccess, canUsePremiumFeature }

useBranding()
  - Retorna logo/watermark efetivo baseado no plano
  - Reage a mudanças no localStorage entre abas

10. INTEGRAÇÃO HOTMART (PAGAMENTOS)
================================================================================

Fluxo:
1. Usuário compra plano Pro ou Premium via Hotmart
2. Hotmart envia webhook POST para /api/webhooks/hotmart
3. API verifica token (hottok) de segurança
4. Extrai email do comprador e ID do produto
5. Resolve qual plano (Pro=7372978, Premium=7393185)
6. Insere/atualiza revela_entitlements no Supabase
7. Quando usuário volta ao app, usePlan() detecta novo plano

Eventos reconhecidos:
  - PURCHASE_APPROVED → ativa plano
  - SUBSCRIPTION_PURCHASE → ativa plano
  - REFUND → revoga plano (volta para free)
  - SUBSCRIPTION_CANCELLATION → revoga plano

Variáveis de ambiente necessárias (Netlify):
  - HOTMART_WEBHOOK_TOKEN (hottok da Hotmart)
  - HOTMART_PRO_PRODUCT_IDS (7372978)
  - HOTMART_PREMIUM_PRODUCT_IDS (7393185)
  - SUPABASE_SERVICE_ROLE_KEY (admin key)


11. INTERNACIONALIZAÇÃO (i18n)
================================================================================

Idiomas: pt-BR (padrão) e en-US

Detecção automática:
  1. localStorage (revela-language)
  2. Timezone brasileiro → pt-BR
  3. navigator.language
  4. Fallback: pt-BR

Arquivo: lib/i18n/translations.ts
  - Interface tipada Translations
  - Seções: home, login, signup, dashboard, projects, settings, common
  - Usado via useLanguage() → { t, language, setLanguage }

Seletor: componente LanguageSelector em header e login/signup

15. SEGURANÇA
================================================================================

- Row Level Security (RLS) no Supabase
- Service Role Key apenas em API routes (servidor)
- Webhook token verification (Hotmart hottok)
- Content-Security-Policy headers
- HSTS (HTTP Strict Transport Security)
- Backup criptografado AES-GCM (client-side)
- Dados locais isolados por email do usuário
- .env.local no .gitignore (segredos não commitados)
